0名词与范围

0.1这是什么 / 替代关系

AI 动态窗口是车机桌面上承载 MVA 语音交互、AI 推荐、应用小窗内容的系统级动态容器。它替代原 FM2.0「小窗」模块:原小窗是有框线、形态固定的窗口;AI 动态窗口无边框、宽度固定、高度随内容动态生长,平时安静待命,被需要时按场景长出恰好的内容。

系统其余部分(状态栏、控制中心、通知中心、地图插件、应用本体等)沿用现有设计,本文档不重复定义,只定义动态窗口与它们的关系。

0.2名词定义

名词定义
MVA语音助手。本文档中 avatar、灵动岛、动态窗口的内容均由 MVA 驱动。
MVA avatar左上角的唤醒头像 / 光球。待机为头像,唤醒时形变为流动光球,代表「从待机到对话」的身份转换。
AI 动态窗口承载内容的那块画布(原型中位于桌面左栏)。无边框、固定宽度、高度向下生长。下设三个互斥视图。
三视图动态窗口的三种互斥内容态:内容态(场景内容 / 对话结果)、推荐流 feed(AI 推荐)、音乐小窗(全屏播放器)。同一时刻只显示其一。
灵动岛底部胶囊之一(AI)。收起为光球,展开为胶囊。是内容态与 feed 的切换入口,也是 AI 推荐的轻量承载。
音乐岛底部胶囊之一。常驻;有歌为胶囊,无歌为球。点开进入音乐小窗。
电话球底部胶囊之一。通话态身份球。
顶栏 ASR语音识别结果与回复文案显示在顶部状态栏,不占用动态窗口内容区。
地图右推动态窗口出内容前,把右侧地图卡向右推开让位(而非覆盖)。

0.3文档范围

本文档只定义三块 1) MVA avatar(左上光球)   2) AI 动态窗口(语音区画布,三视图)   3) 底部三胶囊(灵动岛 / 音乐岛 / 电话球)。
与系统其它窗体的关系(避让、冲突、堆栈、适配)在第 6 章定义;系统其它窗体本身不在范围内。

1总体结构

三块构成与它们在桌面的位置关系如下:

  • MVA avatar —— 左上角。唤醒、身份、思考状态的表达。
  • AI 动态窗口 —— 左栏。无边框画布,承载内容态 / feed / 音乐小窗三视图。出内容时把右侧地图卡推开。
  • 底部三胶囊 —— 底部左侧。灵动岛(AI)、音乐岛、电话球,常驻或按需出现,可并排。

1.1区域分解(四区)

动态窗口整体纵向分为四个区域,各区域尺寸约束与承载内容如下。每个区域内组件的单独状态见 1.3 组件态图。

区域尺寸约束承载内容
状态栏区域锁高不锁宽MVA ASR(识别 / 回复)。见第 2 章。
固定组件区域锁宽不锁高固定组件(原快捷卡片)或思考胶囊,二选一。见 3.3 / 3.5.1。
动态窗口区域锁宽不锁高语音交互内容 / AI 推荐内容 / 应用内容。见 3.3 / 3.5 / 第 7 章。

2MVA avatar

左上角头像是 MVA 的人格载体。它本身不承载内容,只表达状态(待机 / 唤醒 / 思考)。识别文字与回复都落在顶栏,不占动态窗口。

2.1状态

状态表现
待机静态头像。
唤醒 / 形变头像形变为 MVA 光球:旋转的锥形渐变光环 + 白色核心 + 呼吸动画。一个动作完成「待机→对话」的身份转换。
思考复杂任务时,光球进入思考态(呼吸光环加强),配合动态窗口内的思考胶囊(见 3.3)。

2.2与顶栏 ASR 的关系

纳入早期逻辑识别与回复显示在顶部状态栏,不占用动态窗口内容区。
元素行为
识别上屏用户说话时,识别结果在顶栏逐字滚动上屏;识别完成转绿;随后淡出。
回复TTS 回复文案在顶栏显示。回复出现的那一刻,是触发地图右推的时机锚点(见 3.2)。
说明ASR 一律按「识别文字上屏」呈现。设计过程中探索过的「收音光效」方案已废弃,不在本文档。

3AI 动态窗口(语音区画布)

3.1形态

属性定义
边框无边框。靠毛玻璃卡片自身区分层次。
宽度固定。场景越复杂可略宽(原型约 392–424px),由设计 token 锁定。
高度随内容动态向下生长。一行轻量回复到多卡富信息,丰富度随意图自适应。
位置桌面左栏,内容顶部与被推开的地图卡上沿对齐。

3.2地图右推让位 纳入早期逻辑

铁律 · 防叠放遮挡任何动态窗口内容出现之前,必须先把地图卡让开(右推或下移)。
  • 「窗口内容」= 思考胶囊、列表、卡片、规则卡等一切出现在动态窗口里的东西(思考胶囊也算)。
  • 让位动作必须早于或同步先于第一个内容元素渲染,不能等正文出现才推。
  • 时机锚点:放在回复出现的那一刻最稳(识别已结束、内容即将冒出);不要早到「刚识别到文字」就推。
  • 例外:全屏导航底图场景(HUD 叠在地图上是导航应有的),不适用本条。

3.3三视图(互斥)

动态窗口同一时刻只显示一个视图,三者互斥切换:

视图内容进入方式
内容态场景内容 / 对话结果(天气、导航规划、片单、比分、思考胶囊…)。语音交互结果;或从 feed 切回(点灵动岛)。
推荐流 feedAI 推荐流(见 3.4)。点灵动岛收起态光球;或从内容态切换。
音乐小窗全屏播放器(封面、进度、控制)。点音乐岛胶囊主体(见 4.2)。

思考胶囊

规则说明
形态可折叠药丸 + 可展开步骤清单。思考时实时滚动呈现推理 / 检索过程。
收起思考完成后收起为一行结论(可再点开回看)。
工作完成即消失当所属任务完成后(如导航规划已完成扎点),思考胶囊整体消失,不长期占位。
导航中重规划导航态下被重新触发思考时(如「不去接孩子了」),思考胶囊回到顶部;因处于导航态,TBT 卡顺势下移腾位;重规划完成后思考胶囊消失、TBT 复位、点位进度随之更新。

3.4窗口内容类型

对应架构定义,窗口内容分为以下类型:

类型定义状态
空状态无活跃内容。多数时候动态窗口安静(灵动岛仅为光球)。细则待定
AI 推荐(feed)常驻推荐流。见 3.5。已定义
语音交互对话产生的内容态结果。已定义
通知(OTA)通知作为一类窗口内容,可与 feed 共存、并可沉入 feed;多条按优先级聚合;通话 / 音乐进行中被高优通知打断后转入 feed。细则结合 demo 补充。细则待补
应用窗口态(注册制)各应用注册后以窗口态呈现:音乐、电话、PTS、SR、BSV、天气、小程序、手机镜像、CarPlay。见第 7 章。部分待定
内容显示优先级后触发的内容收起当前内容(而非叠加)。具体优先级矩阵待定。

3.5推荐流 feed

规则说明
卡片样式沿用既有 feed 卡(图文混排:图标 / 封面 + 标题 + 副文 + 标签)。
滚动feed 到达高度上限后内部滚动,不向下顶到音乐岛、不遮挡。无「为你推荐」标题。
条数上限feed 总条数封顶(原型为 10),避免无限堆积。
每条可删除每条推荐右侧为删除按钮(✕)(取代原来的 › 箭头);点击以收起动效移除,并同步数据。

3.5.1顶部固定组件(右滑组件库)

规则说明
进入feed 打开时,右滑 feed 区域(或点左侧边缘手柄)→ 滑出组件库编辑面板。
组件库样式参考 iOS 负一屏组件库:显示组件的真实预览(大组件满宽、小组件两列网格),非纯列表。
上限顶部固定区最多 1 大 + 2 小 = 3 个。某尺寸满额后,该尺寸其余组件置灰、不可再加;头部实时计数。已选组件绿色描边 + ✓。
组件清单大组件:天气 / 通勤 / 路线;小组件:日程 / 音乐 / 停车 / 空气 / 续航。
固定区视觉区分固定组件为蓝色玻璃磁贴 + 左侧高亮蓝条,与下方深色推荐行明显区分;固定区与推荐区之间有分割线。固定磁贴高度紧凑。
返回左滑 / 返回后,固定组件置顶显示在 feed 最上方。

3.5.2AI 推荐到达行为

场景行为
feed 在前台在 feed 顶部以动效插入一条新卡(带「刚刚」标 + 短暂高亮)。
feed 隐藏在底部灵动岛胶囊里以通知形式展现;该条同时已进入 feed 数据,用户打开 feed 即在顶部。
待机插入待机时每隔一段时间在 feed 顶部悄悄淡入新推荐,营造实时感而不打扰(总数封顶)。

4底部三胶囊

底部左侧的三个常驻 / 按需胶囊:灵动岛(AI)、音乐岛、电话球。它们位于动态窗口之上,不被内容遮挡,可并排。

4.1灵动岛(AI)

规则说明
两态收起态 = 光球;展开态 = 胶囊。
随机推送模型多数时候灵动岛收起为光球;只有当有 AI 推荐 / 通知进来时才偶尔展开为胶囊;无内容即收起。
作为视图切换入口灵动岛是内容态 ↔ feed 的切换入口。点收起态光球 → 打开 feed;已在某视图时,点击切到另一视图。
软暗示(不用硬按钮)切换不靠「看内容 / 看推荐」硬按钮,而是:① 光球变色 —— 指向 feed 入口时为琥珀核 + 脉动光环,指向内容入口时为青绿核 + 脉动光环;② 胶囊内文案随之变化。
AI 推荐承载feed 隐藏时,新推荐在灵动岛胶囊里以通知形式短暂展现(见 3.5.2)。

4.2音乐岛

规则说明
常驻音乐岛始终可见。
有歌 / 无歌有歌播放 → 胶囊(封面 + 标题跑马灯 + 控制);无歌 → 球。
胶囊样式胶囊较窄,标题文字跑马灯滚动。
暂停 UX点暂停后胶囊保持展开 2 秒再收起;2 秒内再点播放则继续保持展开。
点开音乐小窗点胶囊主体 → 进入音乐小窗(全屏播放器),替换当前内容,是动态窗口的第三视图(全局逻辑)。
与电话互斥通话进行中 → 音乐岛强制收球,不播放音乐(见 4.3)。

4.3电话球

规则说明
通话态身份球通话进行中,以电话球作为该业务的身份胶囊。
音乐让位通话中音乐岛强制收球,通话优先。

4.4多胶囊并排与避让

规则说明
双球并排当存在身份业务(音乐 / 电话)时,底部为两个胶囊并排:AI 球(灵动岛) + 身份球(音乐 / 电话)。
位置关系身份球位于灵动岛胶囊的右边、竖直居中对齐,任何时候不被灵动岛胶囊遮挡。
布局身份球的位置随灵动岛展开 / 收起动态计算(右侧紧贴、垂直对齐)。
待定三者(灵动岛 + 音乐岛 + 电话球)同时存在时的并排次序与间距规则,待补充。

10指标与埋点

本章定义 AI 动态窗口(avatar + 动态窗口 + Smart Bar)的埋点标准,供数据采集与指标分析。事件按本文档模块组织;系统其它部分(应用本体、ASR 引擎等)的埋点由各自模块负责,此处只埋窗口侧的交互。

10.1目标与原则

  • 用户价值可衡量:能回答「灵动岛 / feed / 思考胶囊 是否被用、是否有用」。
  • 全链路可还原:从唤醒 → 视图 → 内容 → 操作 → 转化,事件可串成一次完整交互。
  • 埋点即定义:每个事件有唯一 id、明确触发时机、固定属性;不重复、不歧义。
  • 区分主动与被动:用户主动操作(点击 / 手势)与系统被动呈现(曝光 / 推送)分开记。

10.2通用规范

命名

事件 id 统一前缀 aios_,格式 aios_<模块>_<动作>,小写下划线。事件类型枚举:曝光 / 点击 / 状态 / 手势 / 时长。

公共属性(每条事件都带)

属性说明
event_ts事件时间戳(毫秒)。
session_id会话 id,一个点火 / 使用周期内唯一。
screen屏幕位:csd(主驾)/ psd(副驾)/ rear(副屏)。
screen_size屏幕尺寸:11_9 / 14_4。取值待定
win_view当前窗口视图:hidden / content / feed / music。
island_state灵动岛态:collapsed / expanded。
scene当前场景 / 应用标识(如 weather / nav / music)。
speaker_zone音区:driver / passenger(语音类事件带)。
